# tilmore

아무리 알아도 여전히 더 배울 게 있다.

도움이 됐던 지식을 정리한다. 정리하다 보면 또 배운다.

## 목차

| 섹션 | 설명 |
|------|------|
| [math](math/) | 엔지니어에게 필요한 수학. 미분, 선형대수, 확률, 정보이론, 수치 표현. 수식이 아니라 직관 중심 |
| [systems](systems/) | SIMD, CUDA, 메모리 계층, cache locality, I/O 모델. 성능을 이해하려면 바닥을 알아야 한다 |
| [ai](ai/) | perceptron부터 transformer까지 직접 구현하며 이해하기. 왜 이 구조가 나왔는지 역사와 함께 |
| [production](production/) | 연구와 현실의 격차. 엔지니어링이 연구를 뒤집는 순간들 |
| [architecture](architecture/) | 대규모 시스템 설계. 람다/카파, 배치/스트리밍, 캐시 전략. 트래픽이 커질 때 어떤 선택을 하는가 |
| [scaling](scaling/) | 10 RPS에서 100K RPS까지. 단계별로 아키텍처가 어떻게 진화하는가 |
| [db](db/) | DB 선택의 트레이드오프와 내부 원리. B-tree vs LSM-tree, WAL, MVCC. 왜 그 DB가 그 상황에 맞는지 |
| [distributed](distributed/) | 분산 시스템의 본질. 클럭, 순서, 합의, CAP의 현실적 의미, 장애 전파와 격리 |
| [concurrency](concurrency/) | 동시성의 모델과 함정. lock-free, actor, CSP. 멀티스레드 코드를 왜 틀리게 짜는지 |
| [migration](migration/) | 대규모 마이그레이션 전략. 운영 중인 시스템을 어떻게 바꾸는가 |
| [tools](tools/) | git, tmux, worktree, AI coding tools, 디버깅/프로파일링 도구 |
| [craft](craft/) | 일하는 방식. 코드 리뷰, 추상화 수준, 기술 부채, 설계문서 |
| [leadership](leadership/) | 코드 너머의 역할. 기술 의사결정, 멘토링, 영향력 |
